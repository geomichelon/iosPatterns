Intent

Define an interface for creating an object, but let subclasses decide which class to instantiate.

Problem

Client code needs different variants of a product, but should not depend on concrete types.

Solution

Introduce a factory method that returns a product interface. Concrete factories decide which implementation to create.

Structure

  • The `BadgeShape` protocol defines the product interface.
  • Concrete products implement shape details (circle, square).
  • Factories expose `makeBadge()` and return a `BadgeShape`.

Applicability

  • When product creation varies by configuration or environment.
  • When you want to decouple construction from usage.

SOLID alignment

  • OCP: add new product types via new factories without changing client code.
  • DIP: clients depend on the BadgeShape / BadgeFactory abstractions.

Pros & Cons

Pros

  • Reduces coupling to concrete classes.
  • Creation logic is centralized in factories.

Cons

  • Adds extra types (factories).
  • Small systems may not need the indirection.
