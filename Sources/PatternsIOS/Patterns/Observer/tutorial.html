Intent

Define a one-to-many dependency so when one object changes state, all its dependents are notified.

Problem

Multiple views or services need updates from a central object without tight coupling.

Solution

Introduce an observer interface and a subscription mechanism for registering and notifying observers.

Structure

  • `WeatherStation` acts as the publisher.
  • Observers implement `WeatherObserver`.
  • Updates are broadcast to all subscribers.

Applicability

  • UI updates driven by shared state changes.
  • Event systems and notifications.

SOLID alignment

  • OCP: new observers can be added without changing the subject.
  • DIP: subject and observers depend on the WeatherObserver abstraction.

Pros & Cons

Pros

  • Loose coupling between publisher and subscribers.
  • Dynamic subscription and unsubscription.

Cons

  • Notification order is not guaranteed.
  • Requires careful lifecycle management.
