Intent

Separate the construction of a complex object from its representation so the same process can create different representations.

Problem

A request object has many optional parameters, leading to noisy initializers and errors.

Solution

Use a builder to collect configuration step by step and produce an immutable request at the end.

Structure

  • `RequestBuilder` stores partial configuration with defaults.
  • Fluent setters mutate the builder and return `self`.
  • `build()` produces a `NetworkRequest`.

Applicability

  • Objects with many optional fields.
  • When you want readable, chained configuration.

SOLID alignment

  • SRP: separates construction logic from the final object representation.
  • OCP: new builders/configurations can be introduced without changing client code.

Pros & Cons

Pros

  • Improves readability for complex construction.
  • Keeps the final object immutable.

Cons

  • More types and boilerplate.
  • Overkill for simple objects.
